# 実装ログ・詰まった箇所・原因・改善策

## 1. Next.js プロジェクト作成開始
### 実施内容
- `bun create next-app@latest .` で `my-app` ディレクトリに Next.js アプリを作成しようとした

### 詰まった箇所
- 既存の `README.md` と競合してプロジェクト作成が止まった

### 原因
- `create-next-app` は空ディレクトリ前提で、既存ファイルがあると安全のため中断する

### 改善
- 既存ファイルを削除するか、新しいディレクトリを使う
- 今回は既存ディレクトリを使う方針で進めた

---

## 2. Home画面の初期要件を実装
### 実施内容
- アプリ名を `my-app` とした
- SQLite を `db/my-app.sqlite` に置く方針にした
- `task` テーブルを定義
  - `id`
  - `title`
  - `detail`
  - `status`
  - `updated_at`
  - `created_at`
- Home画面に以下を実装
  - Todoタイトル入力ボックス
  - 新規作成ボタン
  - 未完了タスク一覧
  - 完了タスク一覧

### 詰まった箇所
- SQLite ライブラリ選定で複数回ハマった

---

## 3. `better-sqlite3` の導入失敗
### 実施内容
- 最初は `better-sqlite3` を使って SQLite 接続しようとした

### 詰まった箇所
- 次のようなエラーが発生した
  - `Could not locate the bindings file`
  - `better_sqlite3.node` が見つからない

### 原因
- `better-sqlite3` はネイティブバイナリ依存
- Bun / Next.js / ネイティブ binding の組み合わせで読み込みに失敗した

### 改善
- `better-sqlite3` の利用をやめた

---

## 4. `bun:sqlite` への切り替え失敗
### 実施内容
- Bun なので `bun:sqlite` を使う方針に切り替えた

### 詰まった箇所
- 次のエラーが発生した
  - `Cannot find module 'bun:sqlite'`

### 原因
- Next.js 側のサーバー実行環境で `bun:sqlite` が期待通り扱えなかった
- App Router / Turbopack 側で Bun 組み込みモジュールをそのまま読ませる構成が合っていなかった

### 改善
- `bun:sqlite` も採用をやめた

---

## 5. `node:sqlite` に切り替え
### 実施内容
- `lib/db.ts` で `DatabaseSync` を使う構成へ変更
- DBファイル作成、テーブル自動作成を実装
- `app/page.tsx` でタスク一覧取得・新規作成処理を実装

### 詰まった箇所
- その後、Client Component へタスク一覧を渡した際にエラーが出た

---

## 6. Server Component → Client Component の受け渡しエラー
### 実施内容
- `TaskBoard` を Client Component にして、モーダルや更新機能を実装した

### 詰まった箇所
- 次のエラーが発生した
  - `Only plain objects can be passed to Client Components`

### 原因
- SQLite から取得した行オブジェクトが、そのままだと plain object ではなかった
- Next.js の Server → Client props 制約に引っかかった

### 改善
- `.all()` の返り値を `map()` で明示的に plain object へ詰め直した
- `id`, `title`, `detail`, `status`, `updated_at`, `created_at` を明示変換した

## 11. 新規作成後に一覧が更新されない問題
### 実施内容
- 新規作成フォームを Server Action で実装していた

### 詰まった箇所
- 「新規作成」ボタン押下後、画面に即時反映されず、リロードが必要だった

### 原因
- 作成後にトップページの再検証が走っていなかった

### 改善
- `revalidatePath("/")` を追加して再描画を促した

---

## 12. Hydration warning 発生
### 実施内容
- 画面は表示されていたが hydration warning が出た

### 詰まった箇所
- `data-darkreader-*` 属性による mismatch が出た

### 原因
- Dark Reader 拡張が `<html>` に属性を注入していた
- Server HTML と Client HTML の差分が発生した

### 改善
- アプリロジックの問題ではないと切り分けた
- 必要なら `suppressHydrationWarning` で抑制できる状態にした

---

## 13. Caddy で `/my-app` 配下公開の設定
### 実施内容
- `domain/my-app/` で公開する構成にした
- Caddy で `handle_path /my-app/*` を設定
- `/my-app` → `/my-app/` のリダイレクトも追加した

### 詰まった箇所
- Caddy 起動失敗

### 原因
- `Caddyfile` でコメントと設定が同じ行に壊れて混ざっていた
- `handle` が site block 外にあるように解釈された

### 改善
- `Caddyfile` 全体を整理した
- `sudo caddy fmt`
- `sudo caddy validate`
- `sudo systemctl restart caddy`
の流れで修正した

---

## 14. 本番デプロイシェルの作成
### 実施内容
- `prd-deploy.sh` を作成
- 依存インストール、ビルド、service 再起動、Caddy reload をまとめた

### 詰まった箇所
- `bun install` で integrity check failed
- `next build` 中に OOM でプロセスが kill された

### 原因
- Bun のパッケージ取得検証の不安定さ
- VPS が 1CPU / 1GB と小さく、Next.js build が重かった

### 改善
- Swap を追加した
- 通常デプロイでは毎回 `bun install` をしない方向に整理した
- 最終的には「サーバー上で build しない方がよい」という方針に至った

---

## 15. OOM（メモリ不足）対応
### 実施内容
- `next build` 時にサーバーが不安定化
- OOM Killer により Bun が kill された

### 詰まった箇所
- ビルドが最後まで進まない

### 原因
- 1GB メモリでは Next.js build が重すぎた

### 改善
- Swap を追加した
- サーバー上ビルドは避け、ローカル build + 転送案が有力となった

---

## 16. `bun run dev` 暫定公開を検討
### 実施内容
- build せずに暫定公開したいという方針になった

### 詰まった箇所
- `my-app.service` で `bun run dev` を使うと `node:sqlite` 読み込みが失敗した

### 原因
- アプリは `node:sqlite` を使っており、Bun 実行と相性が悪かった

### 改善
- `bun run dev` をやめた
- `Node.js + next dev` で暫定公開する方針に変更した

---

## 17. systemd での起動失敗 (`203/EXEC`)
### 実施内容
- `my-app.service` を `npx next dev` ベースに書き換えた

### 詰まった箇所
- `status=203/EXEC` で起動失敗

### 原因
- サーバーに `node`, `npm`, `npx` が入っていなかった

### 改善
- `nodejs`, `npm` をインストールした

---

## 18. Node.js バージョン不足
### 実施内容
- `node`, `npm`, `npx` 導入後、service 再起動

### 詰まった箇所
- 次のエラーが出た
  - `Node.js version ">=20.9.0" is required`

### 原因
- Ubuntu 標準の Node.js が `18.19.1` だった
- Next.js 16.1.7 の要件を満たしていなかった

### 改善
- Node.js 22 系を入れる方針に変更
- これにより `next dev` の起動条件を満たした

---

## 19. 暫定公開が動作
### 実施内容
- 最終的に Node.js 経由で my-app が起動
- Caddy 配下で `/my-app/` 公開
- Todo 新規作成、一覧表示、詳細モーダル、編集更新まで動作確認できた

### 結果
- 暫定公開としては動作する状態まで到達した

# 今回の大きな学び

## 1. Bun / Next.js / SQLite の相性は慎重に見るべき
- `better-sqlite3`
- `bun:sqlite`
- `node:sqlite`
のどれを選ぶかでかなり挙動が変わる

## 2. Next.js build は 1GB サーバーではかなり重い
- 本番サーバーで毎回 build は厳しい
- ローカル build + 転送の方が安定する

## 3. 暫定公開なら `next dev` もありだが、本番運用ではない
- 今回は暫定公開として成立
- 本番化するなら `build + start` か `standalone` 化が必要

## 4. Caddy はエラーメッセージをちゃんと読むと直せる
- `validate`
- `fmt`
- `journalctl`
がかなり有効だった
