# myApp デプロイ対応まとめ

## はじめに
myApp を本番サーバーへデプロイする際に直面した課題とその解決策を、PREP 法で整理してまとめました。

私の環境：
- ローカル: Linux Mint 21.2
- 本番サーバー: Ubuntu 22.04
- アプリ：Next.js: v13.4

## 1. ローカル build 方針の確認

### P（結論）
myApp はローカルで build して、本番サーバーへ成果物を転送する方式で進める方針にした。

### R（理由）
Next.js アプリを本番運用する場合、開発サーバー (next dev) をそのまま使うのではなく、build 済み成果物を本番で動かす方が安定するため。

### E（具体例）
既存の別アプリ用デプロイスクリプトを参考にしつつ、myApp では次の成果物を転送する方針を整理した。

- .next/standalone
- .next/static
- public

### P（まとめ）
「ローカル build → 成果物転送 → 本番再起動」という本番向けデプロイ方針を確定した。

### deploy script
```bash
#!/bin/bash
set -e

if [ -z "$SSH_KEY_PATH" ]; then
  echo "❌ SSH_KEY_PATH が未設定です"
  exit 1
fi

if [ -z "$TARGET_INSTANCE" ]; then
  echo "❌ TARGET_INSTANCE が未設定です"
  exit 1
fi

if [ -z "$TARGET_DIR" ]; then
  echo "❌ TARGET_DIR が未設定です"
  exit 1
fi

SSH_KEY_PATH="${SSH_KEY_PATH/#\~/$HOME}"

echo "=== myApp deploy start ==="
echo "TARGET_INSTANCE=$TARGET_INSTANCE"
echo "TARGET_DIR=$TARGET_DIR"

bun run build

ssh -i "$SSH_KEY_PATH" "$TARGET_INSTANCE" "mkdir -p $TARGET_DIR/.next/static $TARGET_DIR/public"

rsync -avz --delete \
  --exclude='db/' \
  --exclude='*.sqlite' \
  --exclude='*.db' \
  -e "ssh -i $SSH_KEY_PATH" \
  .next/standalone/ "$TARGET_INSTANCE:$TARGET_DIR/"

rsync -avz --delete -e "ssh -i $SSH_KEY_PATH" \
  .next/static/ "$TARGET_INSTANCE:$TARGET_DIR/.next/static/"

if [ -d public ]; then
  rsync -avz --delete -e "ssh -i $SSH_KEY_PATH" \
    public/ "$TARGET_INSTANCE:$TARGET_DIR/public/"
fi

ssh -i "$SSH_KEY_PATH" "$TARGET_INSTANCE" \
  "sudo systemctl restart myApp && sudo systemctl status myApp --no-pager"

echo "✅ deploy done"
echo "${MY_DOMAIN_URL}/myApp/"
```

## 2. Linux Mint で build してよいかの確認

### P（結論）
ローカルが macOS ではなく Linux Mint であり、本番が Ubuntu だったため、ローカル build 方式は十分現実的だった。

### R（理由）
OS系統・glibc・CPUアーキテクチャが揃っていれば、Next.js の build 成果物をそのまま持っていける可能性が高いため。

### E（具体例）
確認内容：

- local: glibc 2.39
- server: glibc 2.39
- local: x86_64
- server: x86_64

### P（まとめ）
Linux Mint → Ubuntu の組み合わせなら、ローカル build 方式で問題なく進められると判断した。

---

## 3. node:sqlite による build エラーの解消

### P（結論）
build エラーは Node 本体ではなく @types/node が古かったことが原因だった。

### R（理由）
実行環境の Node は新しかったが、TypeScript の型定義が古く、node:sqlite を認識できなかったため。

### E（具体例）
状態：

- node: v24
- @types/node: v20

これにより import が解決できなかった。

### P（まとめ）
実行環境と型定義のバージョン不一致が原因だった。

---

## 4. standalone build が出ていない問題

### P（結論）
.next/standalone が生成されなかったのは、output: "standalone" が未設定だったため。

### R（理由）
Next.js は明示的に設定しないと standalone 出力を生成しないため。

### E（具体例）
修正：

next.config.ts に output: "standalone" を追加。

### P（まとめ）
standalone 配布には設定が必須だった。

---

## 5. next.config.ts の typo

### P（結論）
build 失敗は単純な typo が原因だった。

### R（理由）
設定ファイルが構文エラーになっていたため。

### E（具体例）
誤：

iimport

正：

import

### P（まとめ）
方向性は正しかったが、単純ミスで build が失敗していた。

---

## 6. rsync デプロイ方式の確立

### P（結論）
デプロイは scp ではなく rsync に統一した。

### R（理由）
差分転送により効率と安定性が高いため。

### E（具体例）
転送対象：

- standalone
- static
- public

### P（まとめ）
ローカル build → rsync → systemd restart の流れを確立した。

---

## 7. TARGET_DIR 未設定問題

### P（結論）
ディレクトリ作成エラーは TARGET_DIR が空だったことが原因。

### R（理由）
空変数によりルート直下へ作成しようとしたため。

### E（具体例）
エラー：

/.next 作成失敗

### P（まとめ）
絶対パスで管理する必要があった。

---

## 8. systemd が dev モードだった問題

### P（結論）
本番なのに next dev で動いていたため修正した。

### R（理由）
開発モードは本番運用に適さないため。

### E（具体例）
変更：

- next dev → server.js
- development → production

### P（まとめ）
本番は standalone + production で動かすべきだった。

---

## 9. CSS / favicon 崩れ

### P（結論）
asset 崩れは basePath 未設定が原因だった。

### R（理由）
サブパス公開時は /myApp 前提にする必要があるため。

### E（具体例）
追加：

basePath: "/myApp"

### P（まとめ）
サブパス公開では basePath が必須だった。

---

## 10. Caddy handle_path 問題

### P（結論）
handle_path により /myApp が削られていたのが原因。

### R（理由）
Next は /myApp 前提で asset を返すため。

### E（具体例）
handle_path → handle に変更。

### P（まとめ）
パスを剥がしてはいけなかった。

---

## 11. 無限リダイレクト

### P（結論）
Caddy と Next の URL 正規化が衝突していた。

### R（理由）
片方が /myApp/、もう片方が /myApp に寄せていたため。

### E（具体例）
ループ：

/myApp → /myApp/ → /myApp

### P（まとめ）
正規URLの統一が必要だった。

---

## 12. トップページ空レスポンス

### P（結論）
真っ黒の原因は HTML が空だったこと。

### R（理由）
Caddy が content-length: 0 を返していたため。

### E（具体例）
外側は空、内側は正常。

### P（まとめ）
Caddy のルーティングが原因だった。

---

# 最終まとめ

## P（結論）
myApp をローカル build → rsync → systemd で本番運用できる状態にした。

## R（理由）
Next.js の本番運用には build・ルーティング・プロキシ整合が必要だったため。

## E（具体例）
対応内容：

- standalone build
- 型エラー解消
- rsync デプロイ
- systemd 修正
- basePath 対応
- Caddy 調整

## P（まとめ）
本番構成への移行が完了し、正常に表示・動作する状態になった。
