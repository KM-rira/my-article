## 私の環境
- ローカル: Linux Mint 21.2
- 本番サーバー: Ubuntu 22.04
- アプリ：Next.js: v13.4

どんな状況だったのか。
local で開発していたときは、APIを呼ぶとDBが更新されているのがわかっていました。
しかし、**本番環境にデプロイしたら、APIは成功しているのに、DBの中身がローカルの情報のままで更新されない**という状況に遭遇しました。
この問題の原因と解決策を、PREP法で整理してまとめました。

今回の記事の結論を先に書いてしまいます。

## Point
今回の問題の本質は、**「更新APIが壊れていた」のではなく、Next.js の表示側が本番DBの最新状態を見ていなかったこと**です。
特に重要なのは、**DBの参照先が違ったのではなく、「ページ表示がキャッシュされていた」こと**です。

---

## Reason
Next.js App Router では、`app/page.tsx` はデフォルトで最適化（キャッシュ・静的化）されます。
そのため、以下のような挙動になります。

- 初回表示時に DB を読み込んでページ生成
- その結果がキャッシュされる
- 以降の表示では DB を再読込せずキャッシュ結果を返す

今回のコードでは、

- `db.prepare(...).all(...)` によって SQLite を直接参照していた
- しかしページ自体がキャッシュされていたため、**DBの最新状態を見に行っていなかった**

つまり、

- API → live DB を更新（成功）
- 画面 → キャッシュ済みの古い描画結果を表示

というズレが発生していました。

---

## Example
今回の調査では、以下の流れで原因を特定しました。

### 1. APIの動作確認
フロント側ログより：

- handleUpdate clicked
- before fetch
- after fetch: 200

これにより、**APIは正常に呼ばれている**と判断。

### 2. 本番DBの確認
SQLiteで確認：

SELECT count(*) FROM task;

結果：

0

しかし画面にはタスク一覧が表示されていたため、
**画面がDBの実態を見ていないことが判明**

### 3. DB参照先の確認
ログより：

DB PATH: /home/ubuntu/repo/growry/db/growry.sqlite

→ **DB参照先は正しい（ズレていない）**

### 4. 表示のキャッシュを疑う
ここで初めて、

- DBは空
- APIは成功
- 画面は更新されない

という矛盾から、**Next.jsのキャッシュ問題と特定**

### 5. 修正
`app/page.tsx` に以下を追加：

export const dynamic = "force-dynamic";

これにより、

- ページが毎リクエスト再実行される
- `db.prepare(...).all(...)` が毎回実行される
- SQLiteの最新状態が表示される

結果：

**画面とDBの状態が一致するようになった**

---

## Point
今回の重要な学びは以下です。

- DBの参照先は最初から正しかった
- 問題は「キャッシュされたページを見ていたこと」
- `dynamic = "force-dynamic"` は
  - DBの参照先を変えるものではない
  - ページの実行タイミングを「毎回」に変える設定

つまり、

**「キャッシュDBを見ていた」のではなく、「live DBを読んだ結果のキャッシュ画面を見ていた」**

というのが正しい理解です。

Next.js App Routerでは、

- APIが正しくても
- 表示がキャッシュされていると
- データが更新されていないように見える

ため、

**API・DB・表示（キャッシュ）の3層で切り分けることが重要**です。

## まとめ

Next.jsのキャッシュ挙動は、特にApp Routerでは注意が必要ですね。
キャッシュって便利ですが、理解がないと「更新されない！」と焦る原因になりますね！
非常にいい経験になりました。今後もこういう問題に遭遇したら、今回の切り分け方法を思い出して対処していきたいと思います。
