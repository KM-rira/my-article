## はじめに

Dockerでアプリをデプロイしていると、こんなことに気づいた。

- GHCRへのpushが異様に遅い
- CI/CDが重い
- imageサイズがやたらデカい

調べてみると原因はシンプルで、

Docker build 時に不要ファイルを全部送り込んでいた

この記事では、
.dockerignore と マルチステージビルドでどう改善したか を実体験ベースでまとめる。

---

## 結論

.dockerignoreを書かないと
・build遅い
・imageデカい
・push遅い
の三重苦になる

マルチステージと組み合わせると一気に改善する

---

## 問題：GHCR push が遅すぎる

自分の環境ではこんな感じだった。

- Next.jsアプリ
- Dockerでbuild → GHCRへpush
- docker-composeで本番反映

なのに…

docker push が毎回めちゃくちゃ遅い（数分〜10分）

---

## 原因①：Dockerは「ディレクトリ丸ごと送る」

ここが一番の罠。

docker build はカレントディレクトリ全体を build context として送る

つまり、

- node_modules
- .git
- .next
- log
- tmp

全部送られていた。

---

## 原因②：不要ファイルがimageに入っていた

例えばこんな状態

node_modules（数百MB）
.git（履歴丸ごと）
.next（ビルド成果物）

→ そのままimageに含まれる
→ GHCR pushが激遅になる

---

## 解決①：.dockerignore を書く

まずはこれ。

```txt
# Git
.git
.gitignore

# Node
node_modules
npm-debug.log

# Next.js
.next
out

# Logs
*.log

# Env
.env
.env.local

# Misc
coverage
dist
tmp
```

これだけで

build context が激減

---

## ポイント：.next を入れるか問題

ここはハマりポイント。

### Docker内で buildする場合（おすすめ）

.next

→ Docker内で build するので不要

---

### ローカル build 済みを使う場合

.next は除外しない

---

## 解決②：マルチステージビルド

次にやったのがこれ。

```docker
# ===== builder =====
FROM node:20-alpine AS builder
WORKDIR /app

COPY package.json package-lock.json ./
RUN npm install

COPY . .
RUN npm run build

# ===== runner =====
FROM node:20-alpine
WORKDIR /app

COPY --from=builder /app/.next .next
COPY --from=builder /app/public public
COPY --from=builder /app/package.json .

RUN npm install --omit=dev

CMD ["node", "server.js"]
```
---

## なぜ効くのか

builder → build用（重くてOK）
runner → 実行用（軽くする）

つまり

- 不要なdev依存 → 消える
- build中のゴミ → 消える
- 最終image → 最小化

---

## 結果

体感だが

Before
- image: 数百MB〜1GB近く
- push: 数分〜10分

After
- image: 数百MB → 数十〜200MB程度
- push: 数分 → 数十秒

---

## さらに重要な気づき

.dockerignore は

「書くと良い」ではなく
「書かないと損する」

---

## よくあるミス

node_modules を含める
→ OS差異で壊れる + 重い

.git を含める
→ セキュリティ的にもNG

.env を消して動かない
→ docker-composeで渡すのが正解

---

## まとめ

・Dockerはディレクトリ丸ごと送る
・.dockerignoreで不要ファイルを除外する
・マルチステージで最終imageを軽くする

この3つで
build速度・imageサイズ・push速度が全部改善する
