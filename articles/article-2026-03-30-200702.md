# PostgreSQL移行 & Docker運用構築まとめ（PREP）

## ■ はじめに（目的・開発環境）

## きかっけ
SQLiteは開発初期には便利だが、スケーラビリティや運用面で限界があるため、PostgreSQLへの移行を決定しました。他のアプリとも共有してDBを利用したいため、外部DB型のPostgreSQLが適していると判断しました。

### やりたいこと
- SQLiteベースのアプリをPostgreSQLへ移行
- Dockerを使った本番運用構成を構築
- アプリとインフラを分離した設計にする
- DB初期化・デプロイを自動化

### 開発環境・ツール
- アプリ：Next.js（App Router）
- DB：PostgreSQL 16
- コンテナ：Docker / Docker Compose
- 言語：TypeScript / Node.js
- DB接続：pg
- 構成：
  - myApp repo（アプリ）
  - infra repo（Docker / インフラ）
- ビルド：multi-stage Docker build
- 実行：standaloneモード（Next.js）

---

# ① DB設計変更（SQLite → PostgreSQL）

## Point
SQLiteの自己初期化型から、PostgreSQLの外部DB型へ移行した。

## Reason
SQLiteはファイルベースでアプリと密結合だが、PostgreSQLは外部サービスのため責務分離が必要。

## Example
問題：
- db.tsにCREATE TABLEを書いていた
- アプリ起動時に毎回DB初期化していた

原因：
- SQLite前提の設計のままだった

改善：
- db.ts → 接続のみ
- init-db.ts → schema管理に分離

## Point
DB接続と初期化を分離することで、運用・拡張性が向上。

---

# ② DATABASE_URLエラー（build時）

## Point
Docker build中にDATABASE_URL未設定でエラーが発生。

## Reason
Next.js build時にdb.tsがimportされ、即時に環境変数チェックが走ったため。

## Example
問題：
- Error: DATABASE_URL is not set

原因：
- build時にはinfraの.envが渡らない
- db.tsで即throwしていた

改善：
- lazy初期化（getPool）に変更
- 接続は実行時のみ行う設計に変更

## Point
環境変数は「build時」と「実行時」で分離して考える必要がある。

---

# ③ Docker ComposeでPostgresが起動しない問題

## Point
pg_isready待機が無限ループになった。

## Reason
postgresコンテナを起動していなかった。

## Example
問題：
- waiting...が無限に続く

原因：
- --no-deps によりpostgres未起動

改善：
- postgresを先に起動
- その後アプリ起動

## Point
依存サービス（DB）は必ず先に起動する必要がある。

---

# ④ tsx not foundエラー

## Point
init-db.ts実行時にtsxが見つからずエラー。

## Reason
runnerコンテナにdevDependenciesが含まれていなかった。

## Example
問題：
- sh: tsx: not found

原因：
- multi-stage buildでdevDependenciesが削除されていた

改善：
- TypeScriptを直接実行せず
- init-db.js（純Node）に変更

## Point
本番環境では「TSを直接実行しない」設計が安定。

---

# ⑤ init-db.jsが見つからない問題

## Point
init-db.jsがコンテナ内に存在せずエラー。

## Reason
Dockerfileのrunnerステージにscriptsがコピーされていなかった。

## Example
問題：
- Cannot find module '/app/scripts/init-db.js'

原因：
- standalone buildは必要最小構成のみコピー
- scriptsディレクトリが除外されていた

改善：
- Dockerfileに以下追加
  COPY --from=builder /app/scripts ./scripts

## Point
multi-stage buildでは「必要ファイルは明示的にコピー」が必須。

---

# ⑥ PostgreSQL接続確認

## Point
コンテナ内のDBに接続し、データ確認を実施。

## Reason
DB初期化が正しく動作しているか確認するため。

## Example
実行：
- psql接続
- \dt
- SELECT * FROM task;

結果：
- テーブル作成確認
- データ確認成功

## Point
DBは必ず「実際にSELECTして確認」することで信頼性が担保される。

---

# ■ まとめ

## Point
SQLiteからPostgreSQLへの移行とDocker運用は成功。

## Reason
責務分離（infra / app）と、build時・実行時の違いを正しく理解できたため。

## Example
今回の重要ポイント：
- DBは外部リソースとして扱う
- buildとruntimeは完全に別物
- Dockerは「何をコピーするか」が重要
- 本番ではTSを直接実行しない

## Point
今回の構成は実務レベルに到達しており、今後はmigration・バックアップ・CI/CDに進める段階。
